Documento de enunciado formal, criterios de evaluación y especificación técnica.

1. Contexto del problema
La universidad cuenta con un servicio interno de salud que ofrece atención en medicina general, psicología, fisioterapia y nutrición. Actualmente la gestión de citas se realiza de forma manual o con herramientas dispersas, lo que genera duplicidad de reservas, uso ineficiente de consultorios, citas en horarios no permitidos y poca trazabilidad de los estados de atención.
La institución requiere una API REST que permita administrar pacientes, doctores, especialidades, consultorios, tipos de cita, horarios de atención y citas médicas, garantizando reglas de negocio reales, control de disponibilidad y reportes operativos básicos.
2. Objetivo general
Diseñar e implementar una aplicación backend con arquitectura por capas, base de datos relacional y pruebas automatizadas, utilizando el stack académico definido para el curso.
3. Objetivos específicos
• Diseñar una aplicación backend con arquitectura por capas usando Java 21, Spring Boot 4, PostgreSQL y pruebas automatizadas.
• Implementar una capa repository con consultas derivadas y JPQL, acompañada por pruebas de integración con Testcontainers.
• Implementar una capa service con reglas de negocio, validaciones de flujo y manejo de conflictos mediante pruebas unitarias con JUnit 5 y Mockito.
• Implementar una capa controller con API REST, validaciones de entrada, manejo de errores y pruebas de controlador con MockMvc.
4. Alcance funcional
• Registrar pacientes, doctores, especialidades, consultorios y tipos de cita.
• Configurar los horarios de atención de los doctores.
• Consultar disponibilidad real por doctor y fecha.
• Crear, confirmar, cancelar, completar o marcar como no asistida una cita.
• Consultar citas por filtros y generar reportes de ocupación, productividad e inasistencia.

Lógica de negocio obligatoria en la capa service
6.1 Reglas para creación de citas
• No se puede crear una cita si el paciente no existe o está inactivo.
• No se puede crear una cita si el doctor no existe o está inactivo.
• No se puede crear una cita si el consultorio no existe o se encuentra inactivo.
• No se puede crear una cita en una fecha y hora pasada.
• La cita debe quedar dentro del horario laboral configurado para el doctor en el día correspondiente.
• El campo endAt no debe llegar calculado por el cliente; el servicio debe construirlo usando la duración del tipo de cita.
• No puede existir traslape de horario para el doctor en el mismo rango temporal.
• No puede existir traslape de horario para el consultorio en el mismo rango temporal.
• Un paciente no puede tener dos citas activas que se crucen en el tiempo.
• Toda cita nueva debe crearse con estado inicial SCHEDULED.
6.2 Reglas de confirmación
• Solo una cita en estado SCHEDULED puede pasar a CONFIRMED.
• No se puede confirmar una cita cancelada, completada o marcada como NO_SHOW.
• La confirmación debe actualizar la fecha de modificación y preservar trazabilidad básica del cambio.
6.3 Reglas de cancelación
• Solo se pueden cancelar citas en estado SCHEDULED o CONFIRMED.
• No se puede cancelar una cita COMPLETED ni una cita NO_SHOW.
• La cancelación debe registrar un motivo obligatorio.
• La cancelación debe liberar inmediatamente el bloque de agenda del doctor y del consultorio.
6.4 Reglas de finalización
• Solo una cita CONFIRMED puede pasar a COMPLETED.
• No se puede completar una cita si la hora actual es anterior al inicio programado.
• Debe permitirse registrar observaciones administrativas al finalizar la cita.
6.5 Reglas de no asistencia
• Solo una cita CONFIRMED puede pasar a NO_SHOW.
• No se puede marcar una cita como NO_SHOW antes de su hora de inicio.
• El sistema debe permitir contar inasistencias por paciente en un período determinado.
6.6 Reglas de disponibilidad y reportes
• La disponibilidad depende del horario laboral del doctor, de las citas existentes y de la duración del tipo de cita.
• La generación de slots debe devolver únicamente bloques completos y libres, no horas aproximadas.
• Debe poder calcularse la ocupación de consultorios por rango de fechas.
• Debe listarse la productividad de doctores por número de citas completadas.
• Deben identificarse pacientes con mayor cantidad de no asistencias.
